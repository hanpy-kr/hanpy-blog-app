# AWS EFS - backup

AWS EFS(Elastic File System)는 고가용성과 높은 내구성을 제공하는 완전관리형 파일 시스템입니다. 다중 AZ에 걸쳐 자동으로 복제되기 때문에 장애에 강하지만, 이것이 곧 백업을 의미하지는 않습니다. EFS는 실수로 인한 파일 삭제, 애플리케이션 오류, 데이터 손상과 같은 상황에 대비한 백업을 기본 기능으로 제공하지 않으며, 운영 환경에서는 별도의 백업 전략이 필요합니다.

---

## AWS Backup을 사용하는 방식

AWS Backup은 EFS를 네이티브로 지원하는 가장 표준적인 백업 방식입니다. 별도의 에이전트나 스크립트 없이 AWS에서 관리형 서비스로 제공되며, 백업 주기와 보존 기간을 정책 형태로 설정할 수 있습니다. 백업 데이터는 Backup Vault에 저장되고, 필요 시 동일 리전 내에서 새로운 EFS 파일 시스템으로 복구할 수 있습니다.

- EFS 네이티브 지원
- 백업 볼트(Backup Vault)에 백업 데이터 저장
- 백업 플랜(Backup Plan)을 통한 스케줄 관리
- 일/주 단위 자동 백업 가능
- 동일 리전 내에서 새 EFS로 복구 가능

운영 부담이 적고 설정이 단순하기 때문에, “백업 관리” 목적이라면 가장 먼저 고려되는 방식이다.

---

## EFS와 EBS

EBS와 달리 EFS는 블록 스토리지가 아니기 때문에 스냅샷 기능을 제공하지 않습니다.

* **EBS**: 볼륨 단위 스냅샷 제공
* **EFS**: 파일 시스템 기반, 스냅샷 없음

EFS 백업은 항상 **파일 단위 백업**이며, 특정 시점의 블록 상태를 그대로 복구하는 방식이 아닙니다. 이러한 구조적 특성 때문에
EFS 백업은 AWS Backup이나 파일 복사 방식을 통해서만 가능합니다..

---

## 비용 관점에서 최소한 알아야 할 점

### AWS Backup 비용

* 백업 스토리지 사용량 기준 과금
* 증분 백업 방식
* 보존 기간이 길수록 비용 증가

### 다른 스토리지로 복사 시 비용

* DataSync 사용 시 데이터 전송 비용
* S3 저장 비용
* 크로스 리전 또는 온프레미스 전송 시 추가 비용 발생 가능

결국 비용 측면에서는 백업 방식보다 보존 정책이 더 큰 영향을 미치므로 내부 서비스 정책을 고려해야합니다.

---

## 정리

EFS는 다중 AZ 기반의 고가용성을 제공하지만, 이는 장애 대응을 위한 구조일 뿐 백업을 의미하지는 않으므로 별도의 백업 전략이 필요하다. 관리형 백업과 복구가 목적이라면 AWS Backup을 사용하는 것이 가장 표준적인 선택이며, 장기 보관이나 데이터 이동이 목적이라면 S3 등 다른 스토리지로 데이터를 복사하는 방식이 적합하다. 또한 EFS에는 스냅샷 기능이 없고, 항상 파일 단위로 백업이 이루어진다는 구조적 특성을 반드시 이해해야 한다.

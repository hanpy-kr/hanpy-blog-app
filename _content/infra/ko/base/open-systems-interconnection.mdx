# OSI(Open Systems Interconnection) 7 Layer

1978년 프랑스의 Hubert Zimmermann이 초안을 발표한 이후, ISO는 1980년 초반에 ‘7계층 참조 모델(OSI Reference Model)’을 제정했습니다. 이 모델은 네트워크 통신 과정을 7개의 계층으로 나누어 하드웨어와 소프트웨어 간 상호운용성을 보장하고자 만든 개념적·추상적 틀입니다. 1984년에는 ISO 공식 표준으로 채택되어 네트워크 프로토콜 개발의 기본 구조로 자리 잡았습니다.

<br />

사실 TCP/IP는 1974년 Vint Cerf와 Bob Kahn의 논문 발표로 시작되어, 1977년 ARPANET 상에서 서로 다른 네트워크를 연결하는 실험을 거쳐 1983년 ARPANET의 공식 프로토콜로 채택되었습니다. 이후 미국 국방성(DoD)이 TCP/IP를 표준으로 채택하고 다양한 운영체제에 구현하면서 빠르게 보급되었습니다.

<br />

OSI 모델은 매우 포괄적이고 이론적인 반면, TCP/IP는 실제 네트워크 환경에 최적화된 단순 구조를 가졌습니다. 1980년대 후반 인터넷이 상용화되면서, 개발자와 제조사들은 구현이 쉽고 성능이 검증된 TCP/IP 스택을 빠르게 도입했습니다. 결과적으로 OSI 기반 제품은 제한적으로만 등장했고, TCP/IP가 사실상의 인터넷 표준으로 자리매김하게 되었습니다. 아래의 글에서는 OSI 모델의 7계층에 대해 알아보도록 합니다.

## 1. 물리 계층(Physical Layer)

물리 계층은 컴퓨터가 주고받는 0과 1의 디지털 데이터를 실제 전기 신호나 전자파로 바꾸는 역할을 합니다. 이 계층에서는 케이블이나 무선처럼 데이터를 물리 매체 위에서 어떻게 보낼지에 대한 전기적·기계적 규격을 정하고, 비트 하나하나를 전압 변화나 주파수 변화 같은 신호로 표현합니다.

<br />

디지털 데이터(0과 1)를 전송하려면 아날로그 신호로 ‘인코딩(Encoding)’하고, 반대로 도착한 신호를 다시 0과 1로 ‘디코딩(Decoding)’해야 합니다. 예를 들어 0은 낮은 전압, 1은 높은 전압으로 바꾸고(또는 특정 주파수 패턴으로 바꾸고), 수신 쪽에서는 그 전압 수준이나 주파수를 보고 다시 0·1로 해석하는 과정입니다. 이렇게 하면 전자기파가 아무리 복잡해도 컴퓨터끼리는 정확하게 같은 이진 데이터로 주고받을 수 있습니다.

<br />

결국 물리 계층은 ‘비트 단위의 전송 규칙’을 정의하는 곳입니다. 이 계층이 있어야 상위 계층에서 “여기서부터 0, 1이구나!”를 정확히 구분할 수 있고, 더 높은 수준의 프로토콜들이 안정적으로 동작할 수 있습니다.

## 2. 데이터 링크 계층(Data Link Layer)

데이터 링크 계층은 네트워크 계층에서 전달된 패킷을 물리 매체로 전송하기 위해 ‘프레임’이라는 단위로 묶는 역할을 합니다. 이 과정에서 데이터를 일정 크기로 나누고, 각 프레임의 시작과 끝을 특별한 비트 패턴(시작·종료 마커)으로 표시해 주어 수신 측이 정확히 어느 지점부터 데이터가 시작되고 끝나는지 인식할 수 있게 합니다. 이렇게 프레이밍된 데이터는 케이블이나 무선 등 물리 매체를 통해 효율적으로 전달됩니다.

<br />

또한 데이터 링크 계층은 프레임에 보낸 사람과 받는 사람의 물리적 주소(MAC 주소)를 헤더에 추가해 네트워크 상에서 목적지를 정확히 지정합니다. 전송 중 데이터가 손상되지 않았는지 검증하기 위해 CRC 같은 오류 검사용 정보를 덧붙이기도 하며, 수신 측에서 이를 검사해 문제가 발생했을 때 재전송을 요청할 수 있습니다. 이처럼 주소 지정과 오류 검출 기능은 물리 계층 단의 단순한 비트 전송만으로는 보장하기 어려운 신뢰성을 높여 줍니다.

<br />

마지막으로 흐름 제어를 통해 전송 속도를 조절하는 것도 데이터 링크 계층의 중요한 역할입니다. 송신 측이 너무 빠르게 데이터를 보내 수신 측의 버퍼가 가득 차 버리는 상황을 방지하기 위해, “지금 잠시 멈춰 달라”는 신호를 주고받으며 적절한 속도를 유지합니다. 이 모든 기능을 통해 데이터 링크 계층은 네트워크 통신에서 ‘안전 벨트’처럼 데이터의 정확성, 순서, 속도를 관리하여 상위 계층이 안정적으로 동작하도록 돕습니다.

<br />

데이터 링크 계층은 프레이밍, 주소 지정, 오류 검출, 흐름 제어의 네 가지 핵심 기능을 통해 원활하고 신뢰성 있는 데이터 전송을 보장합니다. 이를 바탕으로 상위 계층은 데이터가 언제, 어디서, 어떤 순서로 도착하는지를 걱정할 필요 없이 더욱 복잡한 통신 작업에 집중할 수 있습니다.

## 3. 네트워크 계층(Network Layer)

네트워크 계층은 서로 다른 네트워크(예: LAN, WAN)를 연결하여 데이터를 목적지까지 전달하는 역할을 합니다. 이 계층에서는 논리 주소(예: IP 주소)를 사용해 송신자와 수신자를 구분하며, 패킷이 여러 구간을 거쳐 가더라도 출발지와 도착지를 식별할 수 있도록 합니다. 또한 경로 결정(Routing) 기능을 통해 네트워크 토폴로지 정보를 바탕으로 최적의 전송 경로를 찾아 패킷을 보내고, 필요하면 중간 라우터에서 다음 목적지로 전달합니다.

<br />

패킷 크기가 너무 크면 전송 중 손실 위험이 높아지므로, 네트워크 계층은 단편화(Fragmentation)와 재조합(Reassembly) 기능을 제공합니다. 큰 데이터를 MTU(Maximum Transmission Unit)에 맞춰 여러 개의 작은 조각으로 나눈 뒤, 수신 측에서 원래 순서대로 다시 합치는 과정을 거쳐 안정적인 전송을 돕습니다. 이 과정에서 패킷마다 식별자를 붙여 “이 조각들은 같은 데이터의 일부”임을 구분할 수 있게 합니다.

<br />

이제 IP(Internet Protocol)는 네트워크 계층을 구현하는 가장 대표적인 프로토콜입니다. IP는 각 패킷에 헤더를 덧붙여 논리 주소를 저장하고, 비연결형 방식으로 최적 경로를 찾아 전송합니다. 단, IP 자체는 오류 복구나 순서 보장 기능이 없기 때문에, 필요한 경우 TCP 같은 상위 계층 프로토콜을 함께 사용해 신뢰성 있는 통신을 완성합니다.

<br />

네트워크 계층은 논리 주소 지정, 경로 결정, 단편화·재조합을 통해 서로 다른 네트워크를 연결-통제하는 역할을 하며, IP는 이 계층을 구현하는 대표적인 프로토콜이라는 점이 핵심입니다.

## 4. 전송 계층(Transport Layer)

전송 계층은 애플리케이션에서 보내려는 큰 데이터를 여러 개의 작은 조각으로 나누고, 각 조각에 번호(일련번호)를 붙여 전송합니다. 수신 쪽에서는 이 번호를 보고 조각들을 올바른 순서로 다시 조립하고, 전송 중에 손상되거나 빠진 조각이 있으면 “다시 보내 주세요!” 하고 요청할 수 있습니다. 이런 과정을 통해 데이터가 깨지거나 순서가 뒤바뀌지 않도록 지켜 줍니다.

<br />

전송 계층에서 사용하는 대표적인 두 프로토콜이 TCP와 UDP입니다. TCP는 신뢰성을 최우선으로 해서 3-way 핸드셰이크(연결 설정), ACK(확인 응답), 4-way 핸드셰이크(연결 해제) 과정을 거치며 데이터가 순서대로, 빠짐없이 도착하도록 보장합니다. 반면 UDP는 속도를 중시해 이러한 절차를 생략하고 데이터를 가볍게 보내는 방식으로, 실시간 영상·음성 스트리밍이나 온라인 게임처럼 빠른 전송이 중요한 경우에 적합합니다.

<br />

전송 계층은 데이터를 잘게 쪼개고 번호를 매겨 순서대로 안전하게 전달하는 역할을 하며, TCP나 UDP 중 목적에 맞는 프로토콜을 선택해 신뢰성과 속도라는 서로 다른 요구를 충족시킵니다.

## 5. 세션 계층(Session Layer)

세션 계층은 응용 프로그램 간 ‘대화(세션)’를 설정하고 관리하는 역할을 합니다. 예를 들어, 사용자가 웹사이트에 로그인해서 브라우저와 서버가 일정 시간 동안 상호작용할 때, 이 연결을 생성하고 유지하다가 로그아웃 시 종료까지 담당하는 것이 모두 세션 계층의 몫입니다. 이 계층이 없으면 애플리케이션은 “지금 누가 연결돼 있는지?”, “어느 시점까지 대화했는지?”를 알기 어려워 안정적인 통신을 보장하기 힘듭니다.

<br />

세션 계층의 주요 기능은 다음과 같습니다.

1. **연결 설정·유지·종료**: 송·수신 측 간에 대화를 시작할지, 언제 끝낼지 협상합니다.
2. **대화 제어(Dialog Control)**: 전이중(full-duplex), 반이중(half-duplex) 같은 통신 모드를 지정해 “누가 먼저 말할지”를 조율합니다.
3. **동기화(Synchronization) 및 체크포인트**: 긴 트랜잭션이나 파일 전송 중 네트워크 장애가 발생해도, 마지막 체크포인트 이후부터 복구를 시도할 수 있도록 구간별로 ‘저장 지점’을 만듭니다.

한편 OSI 7계층 모델에서 세션(5), 표현(6), 응용(7) 계층은 TCP/IP 모델에서는 하나로 합쳐져 ‘응용 계층(Application Layer)’로 동작합니다. 즉, 실제 인터넷 프로토콜 스택에서는 세션 관리, 데이터 포맷 변환, 애플리케이션 서비스가 모두 상위 계층에서 한꺼번에 처리됩니다.

<br />

세션 계층은 애플리케이션 간 ‘언제’, ‘어떻게’ 대화를 시작하고 끝낼지 관리하며, 대화 중간에 동기화 지점을 두어 안정적이고 효율적인 통신을 가능케 하는 핵심 역할을 합니다.

## 6. 표현 계층(Presentation Layer)

표현 계층은 **애플리케이션 계층**에서 생성된 데이터를 네트워크를 통해 전송하기 전에 송신 측과 수신 측이 “같은 언어”로 이해할 수 있도록 **형식(문법·구조)과 코드(인코딩 방식)를 맞춰 주는** 역할을 합니다. 예를 들어, 한쪽에서는 UTF-8로 인코딩된 텍스트를 보내고 다른 쪽에서는 EBCDIC로 받는다면 의미가 뒤틀리겠죠? 표현 계층이 문자 인코딩(UTF-8, ASCII, ISO-8859 등), 데이터 직렬화(JSON·XML 등) 같은 변환 규칙을 적용해 주면, 애플리케이션끼리 원활하게 주고받을 수 있습니다.

<br />

또한, 표현 계층은 데이터 압축(Compression)과 암호화(Encryption)를 수행합니다.

- **압축/해제(예: ZIP, gzip, JPEG, MPEG)**: 대용량 멀티미디어 파일이나 반복되는 텍스트 데이터는 전송 전에 용량을 줄여 대역폭을 절약하고 속도를 높입니다. 수신 측에서는 다시 원래 상태로 복원(Decompression)하죠.
- **암호화/복호화(예: SSL/TLS, AES)**: 전송 중에 중간에서 도청·변조되지 않도록 데이터를 암호화하고, 수신 측에서는 이를 복호화하여 안전성을 확보합니다.

마지막으로 표현 계층은 **데이터의 표현 방식**을 일관되게 관리함으로써, 애플리케이션 계층이 “내가 받은 데이터가 어떤 구조인지” 고민하지 않아도 되도록 해 줍니다.

<br />

표현 계층은 데이터를 전송하기 전·후에 **인코딩·디코딩**, **압축·해제**, **암호화·복호화** 과정을 책임져, 서로 다른 시스템 간에도 **일관된 형식과 보안**을 유지하도록 돕습니다.

## 7. 응용 계층(Application Layer)

응용 계층은 사용자 애플리케이션과 네트워크 사이를 연결해 주는 가장 상위 계층입니다. 여기서는 우리가 직접 쓰는 웹 브라우저나 이메일 클라이언트, 파일 전송 프로그램 등이 동작하며, 애플리케이션 프로세스가 네트워크 기능을 이용할 수 있도록 지원합니다.

<br />

이 계층에서 사용하는 대표적인 프로토콜로는 HTTP/HTTPS(웹 페이지 송수신), FTP/SFTP(파일 전송), SMTP/POP3/IMAP(이메일 전송·수신), DNS(도메인 ↔ IP 변환), SSH/Telnet(원격 접속) 등이 있습니다. 각각의 프로토콜은 특정 서비스(웹, 메일, 파일, 도메인 조회 등)를 위해 최적화된 규칙을 정의합니다.

<br />

응용 계층은 또한 애플리케이션이 네트워크 기능을 쉽게 호출할 수 있도록 API나 라이브러리 형태의 인터페이스를 제공합니다. 개발자는 복잡한 전송·암호화·세션 관리 과정을 직접 구현할 필요 없이, 간단한 함수 호출만으로도 다양한 네트워크 서비스를 활용할 수 있습니다.

<br />

응용 계층은 “사용자가 체감하는” 네트워크 서비스를 실제로 실행하는 계층으로, 다양한 프로토콜과 개발 인터페이스를 통해 웹, 이메일, 파일 전송 등 최종 애플리케이션 서비스를 가능하게 만듭니다.

## 정리

오늘날 OSI 7계층 모델은 네트워크 교육과 설계 시 참고용 개념 모델로 널리 활용되며, 실제 인터넷 구현은 TCP/IP 모델을 따릅니다. 따라서 OSI의 계층 구조와 TCP/IP의 실무적 프로토콜을 함께 이해함으로써 안정적이고 확장 가능한 네트워크를 구축할 수 있습니다. 조금 더 TCP/IP와 OSI 모델을 비교하자면, TCP/IP 모델과 OSI 모델은 네트워크 통신을 설명하기 위한 구조지만 설계 목적과 사용 방식이 다릅니다. TCP/IP 모델은 실제 인터넷을 설계하고 동작시키기 위해 만들어진 현실적인 구현 모델입니다. 현재 우리가 사용하는 인터넷, 웹, 이메일 등 대부분의 통신은 이 TCP/IP 모델을 따릅니다. 계층 수는 총 4개이며, 응용 계층에서는 HTTP, SMTP 등이 동작하고, 전송 계층에서는 TCP/UDP가, 인터넷 계층에서는 IP, 데이터 링크 및 물리 계층은 하드웨어 기반 네트워크와 연결됩니다. 계층 간 역할이 통합되어 있어 실제 구현에 최적화된 구조입니다. 반면 OSI 모델(Open Systems Interconnection)은 다양한 네트워크 장비와 프로토콜 간 호환성과 이해를 돕기 위한 이론적인 표준 모델입니다. 총 7개의 계층으로 구성되어 있고, 각 계층은 전송, 표현, 세션처럼 세분화된 기능을 담당합니다. 이는 실제 구현보다는 교육, 설계 가이드라인, 프로토콜 분석에 주로 사용됩니다.

<br />

가장 큰 차이는 TCP/IP는 실제 동작하는 인터넷을 위한 구조인 반면, OSI는 이상적인 분리를 통한 이해 중심의 구조라는 점입니다. TCP/IP는 간결하고 현실적이며, OSI는 구조적으로 정밀하고 이상적인 형태를 추구합니다. 두 모델은 목적이 다르기 때문에 계층 수, 구성 방식, 적용 범위에서 차이를 보이며, 결국 TCP/IP는 실제 구현의 표준, OSI는 이론적 기준점으로 함께 이해하는 것이 중요합니다.
